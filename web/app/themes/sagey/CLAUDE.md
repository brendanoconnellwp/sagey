# Sagey — Sage 11 WordPress Theme

## Quick Reference

- Dev server: `npm run dev` (from theme root)
- Production build: `npm run build` (from theme root)
- PHP lint: `./vendor/bin/pint`
- Local URL: http://sageroots.local
- Theme root: `wp-content/themes/sagey/`
- PHP 8.2+ | Node 24+ | Tailwind CSS v4 | Vite 7 | Laravel Blade | Acorn 5

## Critical Rules

1. **ALWAYS** run `npm run build` after modifying any file in `resources/` or any config file
2. **NEVER** edit files in `public/build/` — they are generated by Vite
3. **NEVER** edit the root `theme.json` for runtime styling — it is preprocessed by Vite into `public/build/assets/theme.json`
4. **NEVER** write raw CSS classes. Use Tailwind utility classes. The only CSS files are `resources/css/app.css` and `resources/css/editor.css`
5. **NEVER** echo HTML directly from PHP. Use Blade templates and View Composers
6. **ALWAYS** escape output: `{{ $var }}` for escaped, `{!! $var !!}` ONLY for trusted HTML from WordPress core functions
7. **EVERY** template must `@extends('layouts.app')` and define `@section('content')`
8. **ALWAYS** use closure-based hooks — never named function callbacks
9. **NEVER** add logic to `functions.php` — use `setup.php`, `filters.php`, or create new Providers/Services

## Architecture

### Boot Sequence

```
functions.php
  → require vendor/autoload.php (Composer)
  → Application::configure()
      →withProviders([ThemeServiceProvider::class])
      →boot()
  → loads app/setup.php  (theme supports, menus, sidebars, editor config)
  → loads app/filters.php (WordPress filters)
```

### Stack

| Layer | Technology | Purpose |
|-------|-----------|---------|
| Framework | Acorn v5.1.0 | Laravel container adapted for WordPress |
| Templating | Laravel Blade | Clean template syntax, components, composers |
| Build | Vite 7 + laravel-vite-plugin | Asset compilation, HMR, manifest |
| WP Integration | @roots/vite-plugin | wordpressPlugin (HMR), wordpressThemeJson (theme.json generation) |
| CSS | Tailwind CSS v4 | Utility-first CSS via @tailwindcss/vite (NOT PostCSS) |
| PHP Standards | PSR-4 | Autoloading under `App\` namespace |

### PSR-4 Autoloading

Namespace `App\` maps to the `app/` directory. Any new class placed under `app/` with the correct namespace is autoloaded automatically:

- `App\Providers\ThemeServiceProvider` → `app/Providers/ThemeServiceProvider.php`
- `App\View\Composers\Post` → `app/View/Composers/Post.php`
- `App\PostTypes\Project` → `app/PostTypes/Project.php`

## Cookbook

### Add a Page Template

1. Create `resources/views/template-{name}.blade.php`:

```blade
{{--
  Template Name: My Template
--}}

@extends('layouts.app')

@section('content')
  @while(have_posts()) @php(the_post())
    <div class="mx-auto max-w-3xl px-6 py-16">
      {{-- Your content here --}}
    </div>
  @endwhile
@endsection
```

2. Run `npm run build`
3. Template appears in WP admin Page editor under "Template" dropdown

### Add a Blade Component

1. Create `resources/views/components/{name}.blade.php`:

```blade
@props([
  'variant' => 'default',
])

@php
$classes = match ($variant) {
    'primary' => 'bg-neutral-900 text-white',
    'secondary' => 'bg-neutral-100 text-neutral-900 border border-neutral-200',
    default => 'bg-white text-neutral-900',
};
@endphp

<div {{ $attributes->merge(['class' => $classes]) }}>
  {{ $slot }}
</div>
```

2. Use in any template: `<x-{name} variant="primary">Content</x-{name}>`
3. No PHP class needed for simple components. For complex logic, create `app/View/Components/{Name}.php`

### Add a View Composer

1. Create `app/View/Composers/{Name}.php`:

```php
<?php

namespace App\View\Composers;

use Roots\Acorn\View\Composer;

class Name extends Composer
{
    protected static $views = [
        'partials.your-partial',
        'partials.content-*',
    ];

    public function yourVariable(): string
    {
        return 'value';
    }
}
```

2. Auto-discovered — no registration needed
3. Each public method becomes a template variable: `{{ $yourVariable }}`

### Add a Custom Post Type

1. Create `app/PostTypes/{Name}.php` (create directory if needed):

```php
<?php

namespace App\PostTypes;

class Name
{
    public static function register(): void
    {
        register_post_type('name', [
            'labels' => [
                'name' => __('Names', 'sage'),
                'singular_name' => __('Name', 'sage'),
            ],
            'public' => true,
            'has_archive' => true,
            'show_in_rest' => true,
            'supports' => ['title', 'editor', 'thumbnail'],
            'menu_icon' => 'dashicons-admin-post',
        ]);
    }
}
```

2. Register in `app/setup.php`:

```php
add_action('init', fn () => \App\PostTypes\Name::register());
```

3. Create templates: `resources/views/single-name.blade.php` and/or `resources/views/archive-name.blade.php`
4. Create content partial: `resources/views/partials/content-name.blade.php`

### Add a REST API Endpoint

Register in `app/setup.php` or create `app/Routes/Api.php`:

```php
add_action('rest_api_init', function () {
    register_rest_route('sage/v1', '/example', [
        'methods' => 'GET',
        'callback' => fn (\WP_REST_Request $request): \WP_REST_Response =>
            new \WP_REST_Response(['status' => 'ok']),
        'permission_callback' => fn (): bool => current_user_can('read'),
    ]);
});
```

ALWAYS include a `permission_callback`. Use `'__return_true'` only for truly public endpoints.

### Add a Navigation Menu

In `app/setup.php`, inside the `after_setup_theme` action:

```php
register_nav_menus([
    'primary_navigation' => __('Primary Navigation', 'sage'),
    'footer_navigation' => __('Footer Navigation', 'sage'),
]);
```

Display in a Blade template:

```blade
@if (has_nav_menu('footer_navigation'))
  <nav aria-label="{{ wp_get_nav_menu_name('footer_navigation') }}">
    {!! wp_nav_menu(['theme_location' => 'footer_navigation', 'menu_class' => 'flex gap-6', 'echo' => false]) !!}
  </nav>
@endif
```

### Add Custom Fonts

1. Place font files in `resources/fonts/`
2. Auto-globbed by `resources/js/app.js` via `import.meta.glob`
3. Reference in `resources/css/app.css`:

```css
@font-face {
  font-family: 'CustomFont';
  src: url('../fonts/custom-font.woff2') format('woff2');
  font-weight: 400;
  font-style: normal;
  font-display: swap;
}

@theme {
  --font-custom: 'CustomFont', system-ui, sans-serif;
}
```

4. Use in markup: `class="font-custom"`

### Add Images to Templates

```blade
<img src="{{ Vite::asset('resources/images/logo.svg') }}" alt="Logo" />
```

NEVER hardcode paths to `public/build/`. Always use `Vite::asset()`.

## Blade Syntax Reference

| Pattern | Use |
|---|---|
| `{{ $var }}` | Escaped output (default, safe) |
| `{!! $var !!}` | Unescaped output (ONLY for trusted WP functions) |
| `@php(wp_head())` | Call a WP function inline |
| `@php(the_post())` | Advance the WP loop |
| `@extends('layouts.app')` | Extend master layout |
| `@section('content') ... @endsection` | Define content section |
| `@yield('content')` | Render section (in layout) |
| `@include('partials.name')` | Include a partial |
| `@includeFirst(['partials.content-' . get_post_type(), 'partials.content'])` | Include with fallback |
| `@if / @elseif / @else / @endif` | Conditionals |
| `@while(have_posts()) ... @endwhile` | WordPress loop |
| `@foreach($items as $item) ... @endforeach` | Collection loop |
| `<x-alert type="warning">` | Blade component |
| `@props(['type' => null])` | Component prop declaration |
| `{{ $slot }}` | Component slot content |
| `$attributes->merge(['class' => '...'])` | Merge HTML attributes |
| `@hasSection('sidebar') ... @endif` | Conditional section |

## Tailwind CSS v4 Conventions

### Configuration

Tailwind v4 uses **CSS-based configuration**, NOT `tailwind.config.js`. All customization goes in `resources/css/app.css`:

```css
@import "tailwindcss" theme(static);

/* Content sources for class scanning */
@source "../../app/**/*.php";
@source "../**/*.blade.php";
@source "../**/*.js";

/* Custom design tokens */
@theme {
  --color-brand: #0a0a0a;
  --color-brand-light: #404040;
  --font-heading: 'InterDisplay', system-ui, sans-serif;
  --font-body: 'Inter', system-ui, sans-serif;
}
```

### Rules

- Use Tailwind utility classes for ALL styling — no custom CSS classes
- Use `@theme` to define custom design tokens (colors, fonts, spacing)
- Use `@utility` only for genuinely custom utilities Tailwind cannot express
- Responsive: mobile-first with `sm:`, `md:`, `lg:`, `xl:`, `2xl:`
- Dark mode: `dark:` prefix
- States: `hover:`, `focus-visible:`, `active:`, `group-hover:`
- ALWAYS use `focus-visible:` instead of `focus:` for keyboard-only focus
- Prefer `gap-*` and `space-y-*` over manual margins
- NEVER create a `tailwind.config.js` — that is Tailwind v3 syntax
- NEVER use `@apply` excessively — prefer utilities directly in markup

## Design Philosophy — Modern Minimal

Every page and component must embody **precision, restraint, and clarity**. Think Apple, Linear, Vercel.

### Typography

- Strong visual hierarchy: distinct heading sizes with purposeful weight variation
- Body text: `text-base leading-relaxed tracking-tight` (16px, 1.625 line-height)
- Headings: tight leading (`leading-tight` or `leading-none`), heavier weight (`font-semibold` or `font-bold`)
- Limit to 2 typefaces maximum — one for headings, one for body (or one for both)
- Prefer system fonts or high-quality variable fonts (Inter, Geist, Satoshi)

### Whitespace

- Content must breathe. Generous padding and margins everywhere
- Section spacing: `py-16 md:py-24 lg:py-32` — never less than `py-12`
- Content containers: `px-6 md:px-8` minimum
- Between elements: prefer `space-y-6` or larger over tight stacking

### Color

- Restrained neutral palette: primarily black, white, and grays (`neutral-*`)
- Maximum 1 accent color plus neutrals
- Strong contrast: WCAG AAA preferred (7:1 for body text)
- Subtle borders: `border-neutral-200` light mode, `border-neutral-800` dark mode
- Avoid saturated colors — lean toward muted, desaturated tones

### Motion

- Subtle, purposeful transitions: `transition-colors duration-150` or `transition-all duration-200 ease-out`
- Hover states: subtle opacity or color shifts, never dramatic transforms
- Respect reduced motion: `motion-reduce:transition-none`
- No animations purely for decoration

### Layout

- Content max-width: `max-w-3xl` (48rem, matches theme.json contentSize)
- Wide content: `max-w-5xl` or `max-w-6xl` for galleries, grids
- Full-bleed: `max-w-none` only for hero sections and backgrounds
- Use CSS Grid and Flexbox — never floats
- Consistent alignment — left-align text, center containers

### Images

- Always include descriptive `alt` text
- Use `object-cover` for hero/featured images
- Below-fold images: `loading="lazy"` and explicit `width`/`height`
- Prefer WebP/AVIF formats
- Round corners sparingly: `rounded-lg` or `rounded-xl`, not `rounded-full` on rectangles

### Consistency

- If a pattern appears twice, extract it to a Blade component
- Maintain consistent spacing scale throughout — don't mix arbitrary values
- Use the same border radius, shadow depth, and transition timing across the site

## Accessibility Requirements

- All interactive elements keyboard-accessible
- All images need descriptive `alt` text (empty `alt=""` only for decorative images)
- Color contrast WCAG AA minimum (4.5:1 normal text, 3:1 large text)
- Skip-to-content link in `layouts/app.blade.php` (already exists)
- Semantic HTML: `nav`, `main`, `article`, `section`, `aside`, `header`, `footer`
- ARIA labels on navigation: `aria-label="{{ wp_get_nav_menu_name('primary_navigation') }}"`
- Visible focus indicators: `focus-visible:ring-2 focus-visible:ring-neutral-900 focus-visible:ring-offset-2`
- Forms: every input needs a `<label>`, use `aria-describedby` for help text

## Performance Guardrails

- NEVER load external CSS/JS from CDNs — bundle everything through Vite
- NEVER add `<script>` or `<style>` tags directly in Blade templates
- Images: WebP/AVIF, explicit `width`/`height`, `loading="lazy"` below fold
- Fonts: `font-display: swap`, preload critical fonts, max 2-3 font files
- Keep WP queries efficient: use specific `WP_Query` args, never query all posts
- Minimize PHP logic in templates — move to View Composers
- Database queries belong in Composers, never in templates
- Use `Vite::asset()` for all asset URLs — never hardcode paths

## Security Patterns

### Output Escaping

```blade
{{-- CORRECT: Auto-escaped --}}
{{ $title }}

{{-- CORRECT: Unescaped for trusted WP output --}}
{!! wp_nav_menu([...]) !!}
{!! the_content() !!}

{{-- WRONG: Never unescape user input --}}
{!! $userInput !!}
```

### Nonces

```blade
<form method="POST" action="{{ admin_url('admin-post.php') }}">
  @php(wp_nonce_field('my_action', 'my_nonce'))
  <input type="hidden" name="action" value="my_action">
</form>
```

Verify: `wp_verify_nonce($_POST['my_nonce'] ?? '', 'my_action')`

### Capability Checks

```php
if (! current_user_can('edit_posts')) {
    wp_die('Unauthorized');
}
```

### Data Sanitization

- Input: `sanitize_text_field()`, `absint()`, `sanitize_email()`, `wp_kses_post()`
- Database: `$wpdb->prepare()` for custom queries
- URLs: `esc_url()` for output, `esc_url_raw()` for storage

## Anti-Patterns

1. **NEVER** edit `public/build/` — Vite-generated artifacts
2. **NEVER** add logic to `functions.php` — use `setup.php`, `filters.php`, or Providers
3. **NEVER** write inline `style=""` attributes — use Tailwind classes
4. **NEVER** create custom CSS classes for one-off styling — compose utilities
5. **NEVER** use `<?php echo` in Blade — use `{{ }}` or `{!! !!}`
6. **NEVER** hardcode URLs — use `home_url()`, `get_permalink()`, `Vite::asset()`
7. **NEVER** put database queries in Blade templates — use View Composers
8. **NEVER** use jQuery — use vanilla JS in `resources/js/app.js`
9. **NEVER** skip the WordPress loop (`@while(have_posts()) @php(the_post())`)
10. **NEVER** create `tailwind.config.js` — Tailwind v4 uses CSS config in `app.css`
11. **NEVER** use `@apply` excessively — prefer utility classes in markup
12. **NEVER** ignore accessibility — semantic HTML, ARIA, keyboard nav, contrast
13. **NEVER** use `get_template_directory_uri()` for Vite assets — use `Vite::asset()`
14. **NEVER** use `wp_enqueue_script/style` for theme assets — Vite handles this via `@vite`
